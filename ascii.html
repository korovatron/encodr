PETSCII
PET Standard Code of Information Interchange  ·  Commodore, 1977
Why Commodore built their own encoding
When Commodore designed the PET in 1977, standard ASCII gave them 95 printable characters. That was not enough. The PET had no separate graphics chip — everything on screen, including the user interface, had to be drawn using characters. Commodore needed box-drawing shapes, geometric fills, card suits, and other graphic symbols to make software look good on a character-mapped display. Rather than leave those needs unmet, they replaced a large portion of the ASCII table with their own graphic characters, creating PETSCII. The decision made commercial sense in 1977. The incompatibility cost would compound for decades.
The upside-down alphabet
The most immediately jarring difference: in PETSCII’s default mode, uppercase and lowercase are swapped relative to ASCII. Typing a letter key produces an uppercase letter — just as ASCII specifies. But pressing Shift produces a graphic symbol, not a lowercase letter. Lowercase letters do not exist in the default mode at all. A second character set, switchable at runtime, provides uppercase and lowercase together — but at the cost of losing the graphic characters. This swap was deliberate: Commodore expected most users to work in uppercase with graphics, not to write prose. The PETSCII code for lowercase A (97 in ASCII) maps to a completely different graphic symbol.
Control codes: same bytes, different meanings
The codes 0–31 — ASCII control characters — are reassigned in PETSCII to Commodore-specific screen functions. Code 5 does not send an ENQ; it changes the text colour to white. Code 28 is not File Separator; it switches text to red. Code 18 enables reverse video. The cursor movement keys, function keys, and colour keys all transmit PETSCII control codes in this range. A terminal expecting ASCII control codes receives nonsense. An ASCII terminal expecting CR (code 13) gets it — that one survived — but almost nothing else maps cleanly. The shared byte values with completely different meanings are exactly the kind of incompatibility that makes cross-system communication fail silently rather than visibly.
The graphic characters (96–127 and 160–223)
Where ASCII has punctuation and symbols, PETSCII packs in a set of geometric and graphic characters unique to Commodore hardware: filled and unfilled corners, diagonal lines, cross-hatches, playing card suits (♥ ♦ ♣ ♠), horizontal and vertical bars at various weights, and a pi symbol. These characters were designed to align perfectly with adjacent cells, enabling smooth borders and fills without gaps — exactly what box-drawing characters in CP437 also achieved, but using completely different byte values. A ╚ (CP437 bottom-left double corner, byte 0xC8) is an entirely different character to whatever byte 0xC8 means on a C64. The same visual goal, incompatible implementations.
Real consequences: reading old files today
A .SEQ or .PRG file saved on a Commodore 64 and transferred to a modern PC cannot be opened in a standard text editor and read correctly. Every graphic character appears as a garbled symbol or question mark. Every control code that set a colour or moved the cursor is interpreted as a different control character or printable symbol by the receiving system. To read Commodore files correctly today you need software that loads a PETSCII–to–Unicode mapping table and substitutes each byte value for its correct modern equivalent — exactly the kind of translation layer that Unicode was designed to make unnecessary going forward. This is not a historical curiosity: anyone archiving old Commodore software, converting BASIC listings, or preserving game level data still encounters this problem.
PETSCII’s contribution to the chaos
PETSCII was one of dozens of national and manufacturer variants that fragmented the 1970s–80s computing world. IBM had CP437. Atari had ATASCII. Texas Instruments had their own. Every European country had an ISO 646 variant that replaced certain ASCII symbols with local characters (the UK variant replaced # with £; the French variant replaced @ with à). None of these were compatible with each other. Text that looked correct on the machine it was written on was unreadable elsewhere. PETSCII is a concrete, tangible example of the exact problem Unicode was created to eliminate. Every incompatible encoding was a reasonable local decision. The global cost of all of them together is why a single universal standard eventually became non-negotiable.

What is Unicode?
Unicode is a universal standard that assigns a unique number — called a code point — to every character in every human writing system. U+0041 is always the letter A. U+4E2D is always 中. U+1F600 is always 😀. There are 1,114,112 possible code points (U+000000 to U+10FFFF), of which around 150,000 are currently assigned. A code point is just a number — it says nothing about how to store it as bytes. That is the job of an encoding such as UTF-8.
Why it was introduced
By the late 1980s, the world had hundreds of incompatible 8-bit code pages — CP437, Windows-1252, Latin-1, KOI8-R, Shift-JIS, Big5, GB2312 — each covering a different region. The same byte value meant something completely different depending on which code page the software assumed. A document created on one system displayed as garbage on another. Email and the early web were broken by design. Unicode replaced all of them with a single standard: one number per character, agreed globally, forever.
UTF-8 encoding — how bits are shared between signalling and data
UTF-8 is variable-length: each character uses 1 to 4 bytes depending on its code point. But how does the decoder know how many bytes to read? The leading bits of the first byte carry the length signal — which means fewer bits are available for the actual character data.

A continuation byte always begins with 10, which means it can never be mistaken for a leading byte. Any byte that does not start with 10 is the start of a new character. This lets a decoder instantly resynchronise after a corrupted byte — it simply skips forward until it finds a non-continuation byte.
